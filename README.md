# Online Boutique 微服务智能运维实验项目

本项目为软件测试课程大作业，选择 Online Boutique 开源微服务系统作为实验对象，围绕微服务部署、自动化测试、故障注入、监控采集、异常检测论文复现与智能运维智能体封装完成完整实践。

项目完成第三档要求，并完成两个加分项：

- 第三档：在复杂开源微服务系统 Online Boutique 的基础上完成部署、监控、维护，并进行微服务开发与联调。
- 加分项一：在异常检测与故障诊断相关论文复现基础上，进一步复现并对比 SR-CNN、Donut、FluxEV 三种方法。
- 加分项二：封装智能体自动执行智能运维流程，覆盖故障注入、指标采集、异常检测、根因分析与恢复建议等环节。

## 项目内容

本项目以 Online Boutique 为被测微服务系统。Online Boutique 是一个典型的电商微服务系统，包含前端、商品目录、购物车、结算、支付、推荐、配送、广告等多个服务，适合用于微服务部署、性能测试、监控告警和故障诊断实验。

在原有系统基础上，本项目完成了以下工作：

- 部署 Online Boutique 微服务系统，并保证各服务之间可以正常通信。
- 使用 Selenium 完成 Web 端核心业务流程测试。
- 使用 JMeter 对系统进行压力测试，采集响应时间、请求数、错误率等性能指标。
- 使用 ChaosMesh 进行故障注入，模拟服务异常、资源异常和网络异常等故障场景。
- 使用 Prometheus 采集系统运行指标，并通过 Grafana 进行可视化监控。
- 采集故障注入前后的异常数据，构建 JMeter 与 Prometheus 两类实验数据集。
- 复现 SR-CNN、Donut、FluxEV 三篇异常检测相关论文方法，并在采集数据上进行实验对比。
- 封装智能运维智能体，实现实验流程的自动化组织与结果输出。

## 系统部署与微服务开发

项目选择 Online Boutique 作为实验系统，相比 SockShop，Online Boutique 的服务数量更多、服务调用链更复杂，能够更充分地体现微服务系统在测试、监控和故障诊断中的复杂性。

部署完成后，系统能够支持用户浏览商品、加入购物车、结算支付、查看推荐与配送等核心流程。项目还在原系统基础上进行了微服务开发与接入，使新增或修改的服务能够与原有服务进行交互，并通过测试验证其可用性。

## 项目测试

### Selenium 测试

Selenium 用于模拟真实用户在浏览器中的操作，覆盖 Online Boutique 的核心业务链路，包括：

- 访问前端页面
- 浏览商品列表与商品详情
- 将商品加入购物车
- 修改购物车内容
- 执行结算流程
- 验证页面跳转、响应结果与关键元素状态

Selenium 测试用于验证系统功能正确性和前端业务链路稳定性。

### JMeter 压力测试

JMeter 用于对 Online Boutique 进行并发压力测试，采集系统在不同负载下的性能表现。实验中重点关注以下指标：

- 平均响应时间
- P95 响应时间
- P99 响应时间
- 请求总数
- 成功请求数
- 错误请求数
- 错误率

这些指标同时作为后续异常检测实验中的一类输入数据。

## 故障注入与监控

项目使用 ChaosMesh 对 Online Boutique 进行故障注入，模拟微服务系统中的异常场景。故障注入后，通过 Prometheus 和 Grafana 观察系统指标变化，并采集异常检测所需的时间序列数据。

实验涉及的故障类型包括：

- Pod 故障：模拟服务实例重启、删除或不可用。
- 网络故障：模拟延迟、丢包、网络中断等情况。
- 资源故障：模拟 CPU、内存等资源压力。
- 服务异常：观察关键服务异常对整体调用链的影响。

Prometheus 采集的主要指标包括：

- 节点 CPU
- 节点内存
- Pod CPU
- Pod 内存
- Pod 重启次数
- 服务请求与响应相关指标

Grafana 用于展示系统在正常运行与故障注入期间的状态变化，辅助分析故障传播路径和系统稳定性。

## 异常数据采集与分析

项目采集了两类实验数据：

- JMeter 数据：来自压力测试过程，包含响应时间、请求数、错误率等 7 类指标。
- Prometheus 数据：来自监控系统，包含节点、Pod 和服务层面的 27 类指标。

采集到的数据用于复现和评估 SR-CNN、Donut、FluxEV 三种异常检测算法。实验使用 F1、Precision、Recall 作为主要评价指标。

## 论文复现与实验结果

本项目复现并对比了三种异常检测方法：

- SR-CNN：基于谱残差和卷积神经网络的时间序列异常检测方法。
- Donut：基于 VAE 的无监督时间序列异常检测方法。
- FluxEV：面向微服务指标异常检测与事件分析的方法。

实验结果如下：

| 模型/算法 | 数据集 | F1 | Precision | Recall |
|---|---|---:|---:|---:|
| SR-CNN | JMeter | 0.7549 | 0.6361 | 0.9283 |
| SR-CNN | Prometheus | 0.2135 | 0.1276 | 0.6523 |
| Donut | JMeter | 0.8041 | 0.6860 | 0.9713 |
| Donut | Prometheus | 0.5285 | 0.3625 | 0.9750 |
| FluxEV | JMeter | 0.5665 | 0.7026 | 0.4746 |
| FluxEV | Prometheus  | 0.5700 | 0.8056 | 0.4410 |

从结果可以看出：

- 在 JMeter 数据集上，Donut 的 F1 值最高，整体检测效果最好。
- SR-CNN 在 JMeter 数据集上具有较高召回率，能够发现较多异常点。
- 在 Prometheus 数据集上，FluxEV 的 F1 和 Precision 表现较好，说明其对监控指标类数据有较强适应性。
- Donut 在 Prometheus 数据集上 Recall 较高，但 Precision 相对较低，说明其更倾向于捕获异常，但误报较多。

## 智能运维智能体

项目进一步封装了智能体自动进行智能运维，将故障注入、监控数据采集、异常检测、根因分析和恢复建议组织为自动化流程。

智能体的主要功能包括：

- 自动触发或调度故障注入实验。
- 自动采集 Prometheus 和 JMeter 产生的时间序列数据。
- 调用 SR-CNN、Donut、FluxEV 等算法进行异常检测。
- 根据异常指标、异常时间点和服务拓扑辅助定位可能的异常服务。
- 输出故障分析结果和恢复建议。
- 支持对实验结果进行汇总，便于展示和报告撰写。

该部分对应课程加分项二，体现了从人工观测到自动化智能运维流程的扩展。


## 总结

本项目从微服务系统部署出发，完成了测试、监控、故障注入、异常数据采集、论文复现和智能体封装等完整流程。相比仅复现 SockShop 部署指南，本项目选择了更复杂的 Online Boutique 微服务系统，并在此基础上完成微服务开发、三种异常检测算法复现与智能运维自动化，满足第三档要求及两个加分项要求。
